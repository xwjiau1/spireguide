### 3.2 前端路由设计

### 3.3 核心交互流程

**AI截图问答流程：**
```
1. 用户粘贴/拖拽截图到右侧AI面板
2. 前端上传截图到 /api/ai/recognize
3. 后端转发截图到 Kimi Vision API
4. 返回识别结果（角色、回合、能量、手牌、敌人）
5. 前端展示识别结果，用户可修正/确认
6. 用户确认后，发送 /api/ai/strategy 请求
7. 后端构建结构化Prompt，调用 Kimi Chat API
8. 返回策略建议，前端展示（含"仅供参考"标注）
```

**卡牌查询流程：**
```
1. 用户在搜索框输入关键词
2. 前端调用 /api/cards/search?q=...
3. 后端SQLite全文搜索（LIKE + 索引）
4. 返回匹配卡牌列表
5. 用户点击查看详情（含升级前后对比、配合、流派）
```

## 四、API接口定义

### 4.1 卡牌API

```
GET  /api/cards              # 卡牌列表（支持筛选参数）
GET  /api/cards/:id          # 单张卡牌详情
GET  /api/cards/search?q=... # 搜索卡牌
GET  /api/cards/compare?id1=&id2=  # 卡牌对比
```

### 4.2 敌人API

```
GET  /api/enemies            # 敌人列表
GET  /api/enemies/:id        # 单个敌人详情
GET  /api/enemies/search?q=... # 搜索
```

### 4.3 遗物API

```
GET  /api/relics             # 遗物列表
GET  /api/relics/:id         # 单个遗物详情
```

### 4.4 AI问答API

```
POST /api/ai/recognize       # 截图识别
  Body: { image: File }
  Response: { character, turn, energy, handCards[], enemyName, enemyHp, scene }

POST /api/ai/strategy        # 策略生成
  Body: { 
    question, 
    gameVersion, 
    character, 
    turn, energy, 
    handCards: [{name, cost, effect}], 
    enemyName, enemyHp, enemyIntent,
    confirmed: true 
  }
  Response: { answer, model, tokensUsed }
```

### 4.5 对局记录API

```
GET    /api/sessions         # 对局列表
POST   /api/sessions         # 创建对局
GET    /api/sessions/:id     # 对局详情
POST   /api/sessions/:id/screenshots  # 添加截图
PATCH  /api/sessions/:id/screenshots/:sid  # 更新备注
DELETE /api/sessions/:id/screenshots/:sid  # 删除截图
```

## 五、AI集成方案

### 5.1 截图识别（Kimi Vision）

**Prompt模板：**
```
这是《杀戮尖塔》游戏截图。请识别以下信息并以JSON格式输出：
{
  "scene": "combat|card_reward|map|shop|event|rest|other",
  "character": "角色名（Ironclad/Silent/Defect/Watcher/Necrobinder/Regent）",
  "turn": "回合数",
  "energy": "当前能量/总能量",
  "handCards": [{"name": "卡牌名", "cost": "费用"}],
  "enemyName": "敌人名称",
  "enemyHp": "敌人当前血量/总血量",
  "playerHp": "玩家当前血量/总血量",
  "gold": "金币数"
}
注意：
- 手牌在截图底部区域，每张牌左上角显示费用，中间是名称
- 左下角显示当前能量（蓝色球）
- 上方是敌人区域
- 如果无法识别某字段，填null
```

### 5.2 策略生成（Kimi Chat）

**System Prompt：**
```
你是《杀戮尖塔》攻略助手。基于准确的游戏数据库信息给出策略建议。

规则：
1. 只能使用提供的卡牌效果数据，禁止臆造
2. 解释简洁，新手也能听懂
3. 复杂局势给出2-3个可行方案
4. 不确定时坦诚说明
5. 回答必须用中文
6. 回答末尾必须标注：【仅供参考，请结合实际判断】

当前对局信息：...
```

### 5.3 成本控制

- 图片压缩至 800px 长边再上传
- 相同 image_hash 的问题结果缓存 1 小时
- 匿名用户每日限 10 次问答
- 本地数据增强：识别到手牌名称后，先从数据库查询完整效果再传给AI

## 六、安全与隐私

1. **截图不永久存储**：仅用于AI识别，识别后删除（用户主动保存到对局记录除外）
2. **无登录可用**：支持匿名使用，device_id区分
3. **CORS配置**：仅允许同源访问（生产环境配置Nginx反向代理）
4. **输入校验**：所有API参数用Zod schema校验
5. **SQL注入防护**：使用参数化查询（better-sqlite3自动处理）
6. **Rate Limiting**：express-rate-limit限制API调用频率

## 七、部署方案

### 7.1 开发环境启动

```bash
# 后端
cd backend
npm install
npm run dev          # nodemon + ts-node，端口3001

# 前端
cd frontend
npm install
npm run dev          # Vite dev server，端口5173
```

### 7.2 生产构建

```bash
# 构建脚本（05-构建/build.sh）
cd frontend && npm run build    # 产出 dist/ 目录
cd backend && npm run build     # 产出 dist/ 目录
# 将 frontend/dist 复制到 backend/dist/public
# 启动 backend/dist/index.js
```

### 7.3 服务器部署

- 使用 PM2 管理 Node.js 进程
- Nginx 反向代理 + 静态文件服务
- SQLite 数据库文件持久化到 volume
- 环境变量：KIMI_API_KEY, PORT, NODE_ENV

## 八、已知限制（MVP阶段）

1. **StS2数据不完整**：Early Access持续更新，数据随社区Wiki补充
2. **AI识别精度**：卡牌效果识别依赖模型推断，已通过"人机确认"流程兜底
3. **无用户系统**：匿名使用，无登录态持久化（后续可加）
4. **单机部署**：SQLite单文件，不支持多实例读写（后续可迁移PostgreSQL）
5. **无自动同步**：StS2数据更新需手动触发重新录入（v0.2加入爬虫）
